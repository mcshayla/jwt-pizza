## Response

The on-call engineer was notified of the incident from the Pizza grafana logs and observed the JSONParserErr entries in the Grafana logs panel. The engineer investigated by checking the `/api/docs` endpoint to confirm the factory URL was still correct (`https://pizza-factory.cs329.click`), then identified the API key as the corrupted value. The engineer located the `followLinkToEndChaos` field in the factory's 500 error response by triggering a test order, and used that URL to resolve the chaos. No escalation was needed.

## Root cause

The JWT Pizza Factory chaos system remotely invalidated the factory API key used by the pizza service to authenticate with `https://pizza-factory.cs329.click`. This caused the factory to return a broken/incomplete JSON response instead of a valid pizza JWT, which the service could not parse — resulting in complete pizza order failure.

## Resolution

The service was restored by:
1. Logging in as admin to obtain a fresh auth token
2. Triggering a failing pizza order to receive the HTTP 500 response containing the `followLinkToEndChaos` URL
3. Visiting that URL in a browser to reset the factory API key
4. Confirming resolution via the `{"message":"Problem resolved. Pizza is back on the menu!"}` response

Time to mitigation could be cut in half by adding a Grafana alert that automatically fires when pizza failure metrics spike, and by documenting the `followLinkToEndChaos` resolution process so any team member can resolve it immediately without investigation.

## Prevention


This class of incident (externally injected API key invalidation) could recur any time the chaos system is active. To prevent extended impact: set up a Grafana alert on `pizza_failures_total` metric to page the team immediately when failures begin, and document the `followLinkToEndChaos` resolution URL pattern so it is the first thing checked when factory errors appear.
